## 幻灯片 2：核心发现总结

- mHC（超级连接）主导 Prefill 阶段：占 910C 上 53--84% 的 Prefill 时间；FP32 导致内存瓶颈
- Decode 阶段受 MoE 权重加载限制：路由专家权重加载占 910C Decode 的 38--42%
- H20 Prefill 吞吐量比 910C 高 1.5--2.6 倍，得益于 2.2 倍 HBM 带宽优势
- P/D 分离比例随输入长度递增：8K 为 1P:1D，256K 为 5--8P:1D
- mHC-SP 带来 2.0--4.2 倍 Prefill 加速；910C EP 带宽优势（7.8 倍）支撑高 EP MoE 配置
- **可视化建议**：编号高亮卡片或图标网格，总结六项核心发现

## 幻灯片 3：DeepSeek V4 架构概览

- 43 层 Transformer：2 层全注意力、21 层 C4A（4 倍压缩）、20 层 C128A（128 倍压缩）
- MQA：64 个 Q 头、1 个 KV 头、head_dim=512，Q LoRA 秩 = 1,024
- MoE：256 个路由专家（top-6 路由）、1 个共享专家、inter_dim=2,048
- mHC（超级连接）：每个子层均有 FP32 前/后投影 + Sinkhorn 归一化
- Lightning Index：64 个索引头、dim=128、topK=512，用于压缩 KV 条目选择

## 幻灯片 4：硬件对比 — 昇腾 910C vs NVIDIA H20

- Cube 算力（BF16）：910C = 376 TFLOPS，H20 = 148 TFLOPS（910C 2.54 倍优势）
- HBM 带宽：910C = 1,800 GB/s，H20 = 4,000 GB/s（H20 2.22 倍优势）
- EP 带宽：910C = 392 GB/s，H20 = 50 GB/s（910C 7.84 倍优势）
- HBM 容量：910C = 64 GB，H20 = 96 GB（H20 1.5 倍优势）
- Cube:Vec 比：910C = 15.7:1（不均衡），H20 = 3.4:1（更均衡）

## 幻灯片 5：方法论 — Roofline 性能模型

- 每个算子由三个资源维度刻画：Cube 时间、Vec 时间、Mem 时间
- 瓶颈 = argmax(cube, vec, mem)；总时间 = max(cube, vec, mem) + 通信时间
- 浮点运算约定：矩阵乘法 [M,K] x [K,N] = M x N x K x 2；BF16 = 2 字节，FP32 = 4 字节
- 利用率假设：计算 50%，HBM 带宽 80%（两个平台一致）
- 搜索空间：TP 属于 {1..64}，EP 属于 {1..256}，DP 属于 {1..8}，BS 属于 {1..512}；约束：(TP x DP) % EP == 0
- **可视化建议**：Roofline 图，展示算力 vs 访存强度，标注示例算子位置

## 幻灯片 6：参数搜索结果 — 8K/4K（双平台）

- 910C Prefill 吞吐：404.5 tps/gpu（TP=8, EP=16, DP=2, BS=256, 16 GPUs）
- H20 Prefill 吞吐：1,040.2 tps/gpu（TP=8, EP=8, DP=1, BS=128, 8 GPUs）— 高 2.57 倍
- 910C Decode 吞吐：135.3 tps/gpu（TP=4, EP=16, DP=4, BS=512, 16 GPUs）
- H20 Decode 吞吐：207.7 tps/gpu（TP=8, EP=16, DP=2, BS=512, 16 GPUs）— 高 1.53 倍
- 910C Decode 偏好 TP=4（访存瓶颈），H20 仅用 8 GPUs 即可达到最优 Prefill 吞吐

## 幻灯片 7：参数搜索结果 — 128K/4K（双平台）

- 910C Prefill 吞吐：314.0 tps/gpu（16 GPUs）；H20：597.4 tps/gpu（8 GPUs）— H20 高 1.90 倍
- 910C Decode 吞吐：15.7 tps/gpu（32 GPUs）；H20：45.6 tps/gpu（16 GPUs）— H20 高 2.91 倍

- 910C Prefill 延迟：26.1s；H20：27.3s — 基本持平（910C 算力优势抵消 H20 带宽优势）
- 910C Decode 延迟：21.2 ms/step；H20：9.9 ms/step — H20 快 2.1 倍

## 幻灯片 8：参数搜索结果 — 256K/4K（双平台）

- 910C Prefill 吞吐：253.5 tps/gpu（16 GPUs）；H20：410.9 tps/gpu（8 GPUs）— H20 高 1.62 倍
- 910C Decode 吞吐：8.3 tps/gpu（32 GPUs）；H20：24.6 tps/gpu（16 GPUs）— H20 高 2.98 倍
- 910C Prefill 延迟：64.5s；H20：79.5s — 910C 快 1.23 倍（二次注意力下 Cube 算力优势显现）
- 最大 batch 再次减半：910C 上 BS=8，H20（EP=8）上 BS=4
- 256K 服务在两个平台上都代价高昂，需考虑分块策略

## 幻灯片 9：长上下文扩展趋势

- Prefill 延迟超线性增长：910C 上 1.3s -> 26.1s -> 64.5s（8K -> 128K -> 256K）
- Decode 吞吐急剧下降：910C 上 135.3 -> 15.7 -> 8.3 tps/gpu（8K -> 128K -> 256K）
- Decode 延迟相对稳定：19.4 -> 21.2 -> 21.7 ms/step（压缩机制有效控制注意力开销）
- H20 在长序列 Prefill 优势缩小（910C Cube 算力在二次注意力中发挥作用）
- H20 在长序列 Decode 优势扩大（更多 KV 缓存读取受益于 2.2 倍 HBM 带宽）

## 幻灯片 10：P/D 分离服务概念

- Prefill/Decode 分离服务：P 实例处理输入 token（Prefill 阶段），D 实例生成输出 token（Decode 阶段）
- QPS 平衡方程：N_p x P_rps >= N_d x D_rps
- P:D 比例公式：N_p/N_d >= (D_tps_instance x input_len) / (P_tps_instance x output_len)
- 比例由输入/输出长度不平衡驱动：8K/4K 为 2:1 -> 256K/4K 为 64:1
- 混合（非分离）服务在长上下文下浪费 60--80% 的 GPU 资源
- **可视化建议**：概念图，展示 P 实例向 D 实例输送请求的 QPS 流向箭头

## 幻灯片 11：P/D 比例结果表

- 910C 8K/4K：1P:1D（P=16 GPUs，D=16 GPUs）— Prefill 和 Decode 平衡
- 910C 128K/4K：4P:1D（P=4x16 GPUs，D=1x32 GPUs）— Prefill 成为瓶颈
- 910C 256K/4K：5P:1D（P=5x16 GPUs，D=1x32 GPUs）— 更加偏重 Prefill
- H20 128K/4K：5P:1D；256K/4K：8P:1D — H20 需要更多 P 实例（D 更快，P 吞吐相近）
- 长上下文下 910C 所需 P 实例少于 H20（D 较慢部分平衡了比例）

## 幻灯片 13：SP 与 mHC-SP 对比

- 三种配置测试：无 SP（基线）、仅 SP、SP + mHC-SP
- 仅 SP：910C 上提升有限（6--9%），H20 上提升显著（27--48%），因减少了 EP AllToAll 开销
- 910C 8K 下 mHC-SP：mHC 占比从 84% 降至 40%（整体加速 4.19 倍）
- mHC-SP 收益随序列长度递减：8K 为 4.2 倍，128K 为 2.5 倍，256K 为 2.0 倍
- mHC-SP 原理：将 FP32 mHC 操作在 TP 维度并行化（TP=8 -> mHC 减少约 8 倍）
- **可视化建议**：分组柱状图，展示三种配置下 8K/128K/256K 的 Prefill 时间（ms）

## 幻灯片 14：mHC-SP 加速分析

- 910C 加速倍数（无 SP -> SP+mHC-SP）：4.19x（8K）、2.53x（128K）、1.98x（256K）
- H20 加速倍数：3.39x（8K）、2.04x（128K）、1.65x（256K）
- 加速递减原因：二次注意力计算增长，不受 mHC-SP 影响
- mHC-SP 后瓶颈转移至注意力投影（23.6%）和通信（17.4%）（8K 场景）
- 长上下文场景下，注意力优化比 mHC 优化更为重要

## 幻灯片 15：算子瓶颈分析 — Prefill 阶段

- 注意力投影：CUBE 瓶颈（计算密集型矩阵乘法）
- 注意力计算：CUBE 瓶颈（随序列长度二次增长）
- mHC 前/后投影：MEM 瓶颈，即使在 Prefill 阶段（FP32 使内存流量翻倍）
- Sinkhorn：910C 上 VEC 瓶颈（纯 FP32 向量运算，无矩阵乘法并行性）
- MoE 门控和路由专家：Prefill 阶段 CUBE 瓶颈
- **可视化建议**：表格，按类别标注瓶颈类型（CUBE / VEC / MEM / COMM），颜色编码

---

## 幻灯片 16：算子瓶颈分析 — Decode 阶段

- MoE 路由专家：MEM 瓶颈（权重加载主导，占 910C Decode 的 38--42%）
- 注意力投影：MEM 瓶颈（单 token Decode -> 低算术强度）
- 注意力计算：MEM 瓶颈（KV 缓存读取主导）
- mHC：MEM 瓶颈（FP32，与 Prefill 相同但占比较小）
- 通信：COMM 瓶颈（MoE 的 AllToAll，注意力的 AllReduce）
- **可视化建议**：表格，按类别标注瓶颈类型；与 Prefill 幻灯片并排对比

## 幻灯片 18：910C vs H20 对比总结

- Prefill 吞吐：H20 胜出 1.62--2.57 倍（HBM 带宽缓解 mHC 内存瓶颈）
- Decode 延迟：H20 稳定胜出约 2 倍（直接反映 2.2 倍 HBM 带宽比）
- 910C 在 256K Prefill 延迟上具有优势（Cube 算力助力二次注意力）
- 910C EP 带宽（7.84 倍）支持 EP=32/64 配置，H20 上代价过高
- H20 所需总 GPU 数始终更少（24--80）vs 910C（32--112），用于平衡 P/D 服务

## 幻灯片 19：行业启示与优化机会

- KV 压缩（C4A/C128A）不可或缺：无压缩时 128K 上下文在合理 batch 下不可行
- mHC 优化是最高 ROI 目标：mHC-SP（2--4 倍）、FP16/BF16 mHC、融合 Sinkhorn 算子
- Cube:Vec 比影响显著：910C 的 15.7:1 导致 FP32 归一化工作负载下 vec/mem 严重闲置
- HBM 带宽是 Decode 阶段的关键差异化因素：对话类负载中 HBM 带宽 > 计算 TFLOPS
- MoE 模型强烈依赖高带宽互联：NVLink 域内高效，跨节点 MoE 代价高昂
- **可视化建议**：矩阵图，将优化机会映射到影响程度（高/中/低）和实施难度

## 幻灯片 20：部署建议

- 8K/4K（对话/编程）：910C = 32 GPUs（1P:1D），H20 = 24 GPUs（1P:1D）；H20 更具成本效益
- 128K/4K（RAG/文档问答）：910C = 96 GPUs（4P:1D），H20 = 56 GPUs（5P:1D）；P/D 分离必不可少
- 256K/4K（全文档分析）：910C = 112 GPUs（5P:1D），H20 = 80 GPUs（8P:1D）；考虑分块替代方案
- 通用建议：始终启用 SP，研究 mHC-SP 可行性，按网络调优 EP（910C 高 EP，H20 低 EP）
- Decode 阶段积极增大 batch（吞吐随 batch 线性增长直至内存上限）；128K+ 必须采用 P/D 分离
- **可视化建议**：决策流程图或按使用场景分类的推荐配置表，包含平台特定参数
