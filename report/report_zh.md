# DeepSeek V4 推理性能分析：Ascend 910C vs NVIDIA H20

## 1. 摘要

本报告基于 Roofline 模型，对 DeepSeek V4 在两种硬件平台（Ascend 910C 和 NVIDIA H20）及四种服务场景（8K/4K、32K/4K、128K/4K、256K/4K 输入/输出 token 长度）下的推理性能进行了全面分析。主要发现如下：

1. **mHC 内核融合（默认开启）将 Prefill 时间缩短 3--4 倍**，通过消除 HBM 流量。融合后，mHC 在 910C 上 8K 场景下从 Prefill 时间的 84% 降至 36%。
2. **910C 在 Prefill 吞吐量上接近持平** —— 内核融合消除了 mHC 内存瓶颈（H20 的带宽优势在此占主导地位），差距从 2.57 倍缩小至 8K 场景下的仅 1.12 倍。
3. **Decode 在 910C 上受限于 MoE 权重加载（40--56%）**，在 H20 上短上下文时受限于通信（54%），长上下文时转为 MoE 权重加载受限。
4. **V4 的 KV 压缩相比 V3 节省 4.6 倍内存** —— 使长上下文服务在实际中可行，而 V3 的 MLA 方案无法匹配。
5. **P/D 分离比例随上下文长度增长**：910C 上为 1P:1D（8K）、1P:1D（32K）、2P:1D（128K）、3P:1D（256K）。
6. **910C 的 EP 带宽优势（7.8 倍）** 支持高 EP 的 MoE 配置，而这在 H20 上成本过高。

**分析方法**：Roofline 模型 —— 每个算子时间 = max(cube, vec, mem) + comm。利用率：50% 计算，80% HBM 带宽。

## 2. 模型架构

### 2.1 DeepSeek V4 架构

DeepSeek V4 是一种混合专家模型（Mixture-of-Experts, MoE），包含 43 个 Transformer 层：
- **2 个全注意力层**（ratio=1）：标准 MQA，64 个 Q 头，1 个 KV 头
- **21 个 C4A 层**（ratio=4）：4 倍 KV 压缩，配合 Lightning Index（topK=512）
- **20 个 C128A 层**（ratio=128）：128 倍 KV 压缩，无需索引

关键架构特征：
- **MQA**（Multi-Query Attention）：64 个 Q 头，1 个 KV 头，head_dim=512
- **Lightning Index**：64 个索引头，dim=128，选择 top-512 压缩 KV 条目
- **MoE**：256 个路由专家（top-6），1 个共享专家，inter_dim=2048
- **mHC（Hyper Connection）**：每个子层均有 FP32 前/后投影和 Sinkhorn 归一化
- **KV 压缩**：分组投影按比例压缩 K 和 V 缓存

| 参数 | 值 |
|---|---|
| Hidden size | 4,096 |
| Layers | 43 (2 full + 21 C4A + 20 C128A) |
| Q heads | 64 (MQA: 1 KV head) |
| Head dim | 512 |
| Q LoRA rank | 1,024 |
| O groups | 8, O LoRA rank = 1,024 |
| Index heads | 64, dim = 128, topK = 512 |
| Window size | 128 (SWA) |
| Routed experts | 256, top-6 |
| Shared experts | 1 |
| MoE inter dim | 2,048 |
| HC mult | 4 |
| Vocab size | 129,280 |

### 2.2 V4 与 V3 对比

| 维度 | DeepSeek V4 | DeepSeek V3 |
|---|---|---|
| Hidden size | 4,096 | 7,168 |
| Layers | 43 | 61 |
| Q heads | 64 | 128 |
| KV 方案 | MQA + KV compression | MLA (kv_lora_rank=512) |
| Q LoRA rank | 1,024 | 1,536 |
| Routed experts | 256, top-6 | 256, top-8 |
| MoE inter dim | 2,048 | 2,048 |
| KV compression | C4A/C128A (2--128x) | 无 |
| mHC (Hyper Connection) | 有 (hc_mult=4) | 无 |
| Lightning Index | 有 (64 heads, dim=128) | 无 |
| 总参数量（约） | ~286B | ~704B |
| 每 token KV 缓存 | 15,168 bytes | 70,272 bytes |
| **KV 压缩比** | **比 V3 少 4.6 倍** | 基线 |

V4 以 V3 较大的隐藏维度为代价，换取了激进的 KV 压缩和新型架构特征（mHC、Lightning Index）。4.6 倍的 KV 缓存节省对于实际长上下文服务至关重要。

### 2.3 硬件平台

| 指标 | Ascend 910C | NVIDIA H20 | 比率 |
|---|---|---|---|
| Cube TFLOPS (BF16) | 376 | 148 | **910C 2.54x** |
| Vec TFLOPS (FP32) | 24 | 44 | **H20 1.83x** |
| Cube:Vec 比率 | 15.7:1 | 3.4:1 | H20 更均衡 |
| HBM 带宽 (GB/s) | 1,800 | 4,000 | **H20 2.22x** |
| HBM 容量 (GB) | 64 | 96 | **H20 1.5x** |

最显著的硬件差异：910C 拥有 **7.8 倍的 EP 带宽**（节点内互联），但 **HBM 带宽低 2.2 倍**。

## 3. 瓶颈分析

### 3.1 各类别瓶颈总结

| 类别 | Prefill 瓶颈 | Decode 瓶颈 |
|---|---|---|
| 注意力投影 | CUBE | MEM |
| 注意力计算 | CUBE | MEM |
| KV 压缩 | CUBE | VEC/MEM |
| Lightning Index | COMM（score AllReduce） | COMM |
| mHC（融合） | MEM | MEM |
| MoE 门控 | CUBE | MEM |
| MoE 路由专家 | CUBE | MEM |
| 通信 | COMM | COMM |

**关键洞察：Prefill 受限于计算（CUBE），Decode 受限于内存（MEM/COMM）。**

### 3.2 Prefill 算子分布（910C，最优吞吐量配置，融合）

| 类别 | 8K/4K | 32K/4K | 128K/4K | 256K/4K |
|---|---|---|---|---|
| mHC | **36.3%** | **29.4%** | **16.7%** | **10.6%** |
| 注意力投影 | 25.4% | 20.6% | 11.7% | 7.4% |
| 注意力计算 | 7.8% | 16.2% | 31.6% | 39.0% |
| Lightning Index | 4.8% | 13.1% | 28.2% | 35.5% |
| 通信 | 18.1% | 14.6% | 8.3% | 5.3% |
| KV 压缩 | 2.4% | 2.0% | 1.1% | 0.7% |
| MoE（全部） | 3.0% | 2.4% | 1.4% | 0.9% |
| 其他 | 2.2% | 1.8% | 1.0% | 0.6% |

在 8K 时，mHC（内存受限）是最大的单一类别。随着序列长度增加，注意力计算和 Lightning Index 以二次方增长并占据主导。

### 3.3 Decode 算子分布（910C，最优吞吐量配置）

| 类别 | 8K/4K | 32K/4K | 128K/4K | 256K/4K |
|---|---|---|---|---|
| MoE 路由 | **55.9%** | **38.1%** | **40.3%** | **41.1%** |
| 注意力计算 | 14.5% | 25.2% | 22.0% | 21.4% |
| 通信 | 12.9% | 14.4% | 12.4% | 12.2% |
| Lightning Index | 5.7% | 11.3% | 16.4% | 16.7% |
| 注意力投影 | 5.4% | 6.8% | 6.8% | 6.9% |
| mHC | 4.1% | 2.8% | 0.7% | 0.4% |
| 其他 | 1.5% | 1.4% | 1.4% | 1.3% |

Decode 由 MoE 权重加载主导（38--56%）。随着序列长度增加，注意力和 Lightning Index KV 缓存读取占比增长。

### 3.4 910C vs H20 瓶颈对比

**Prefill 瓶颈差异显著：**

| 类别 | 910C 8K | H20 8K | 根本原因 |
|---|---|---|---|
| mHC | 36.3% | 9.1% | 910C 较低的 HBM 带宽使 MEM 受限的 mHC 更严重 |
| 注意力投影 | 25.4% | 36.0% | H20 较低的 Cube TFLOPS 使瓶颈转向 CUBE 算子 |
| 通信 | 18.1% | 26.6% | H20 低 EP 带宽（50 vs 392 GB/s） |

**Decode 瓶颈也存在差异：**

| 类别 | 910C 8K | H20 8K | 根本原因 |
|---|---|---|---|
| MoE 路由 | 55.9% | 17.4% | 910C 较低的 HBM 带宽使权重加载占据主导 |
| 通信 | 12.9% | 54.2% | H20 低 EP 带宽主导 Decode |

在 910C 上，Decode 受限于权重加载。在 H20 短上下文场景下，Decode 受限于通信，因为 EP AllToAll 开销高昂（50 GB/s）。在长上下文（128K+）时，两个平台均转为 MoE 权重加载受限。

#### Prefill 与 Decode 延迟

| 场景 | 910C Prefill (ms) | H20 Prefill (ms) | 910C Decode (ms) | H20 Decode (ms) |
|---|---|---|---|---|
| 8K/4K | 330 | 554 | 19.3 | 9.0 |
| 32K/4K | 1,535 | 2,771 | 19.4 | 9.1 |
| 128K/4K | 10,747 | 20,396 | 21.0 | 9.9 |
| 256K/4K | 33,923 | 65,685 | 21.6 | 10.1 |

910C 在所有序列长度下实现 **1.7--1.9 倍的 Prefill 延迟优势**（Cube TFLOPS 优势）。H20 稳定实现 **约 2 倍的 Decode 延迟优势**（HBM 带宽优势）。

#### 根因总结

**910C 的优势场景：**
- Cube 密集型工作负载：Prefill 中的注意力投影、MoE 专家矩阵乘法
- 高 EP 配置：EP=32/64 成本较低，得益于 392 GB/s 的 EP 带宽
- Prefill 延迟：在所有序列长度下比 H20 快 1.7--1.9 倍

**H20 的优势场景：**
- 内存受限工作负载：Decode 权重加载、KV 缓存读取
- 低 EP 吞吐量：EP=8 仅用 8 个 GPU 即可匹配或超过 910C 的 16 GPU 配置
- Decode 阶段：始终快约 2 倍，长上下文时扩大至 3 倍

**H20 的 EP 带宽问题：** H20 的 50 GB/s 跨节点 EP 带宽（对比 910C 的 392 GB/s）迫使其在吞吐量场景中使用 EP=8，每 rank 需承载 32 个专家。H20 的 96 GB HBM 可以容纳，但限制了扩展性。

## 5. 关键模块分析

### 5.1 mHC 优化

#### 优化级别

| 级别 | mhc_kernel_fused | mhc_sp | mhc_fused_bf16 | 描述 |
|---|---|---|---|---|
| 未融合 FP32 | False | False | False | 原始基线 |
| 融合 FP32 | True | False | False | 内核融合（新默认值） |
| 融合 FP32 + SP | True | True | False | + mHC 序列并行 |
| 融合 BF16 + SP | True | True | True | + 融合算子采用 BF16 精度 |

#### Prefill 时间对比（910C，TP=8，EP=16，DP=2，BS=16）

| 级别 | 8K/4K | mHC % | 32K/4K | mHC % | 相对未融合加速比（8K） |
|---|---|---|---|---|---|
| 未融合 FP32 | 10,144 ms | 84.3% | 42,853 ms | 79.8% | 1.00x |
| 融合 FP32 | 2,492 ms | 36.0% | 12,244 ms | 29.3% | 4.07x |

| 融合 BF16 + SP | 1,650 ms | 3.4% | 8,877 ms | 2.5% | 6.15x |

#### 瓶颈迁移（910C，8K/4K）

| 类别 | 未融合 FP32 | 融合 FP32 | 融合+SP | 融合 BF16+SP |
|---|---|---|---|---|
| **mHC** | **84.3%** | **36.0%** | **6.6%** | **3.4%** |
| 注意力投影 | 6.2% | 25.2% | 36.9% | 38.1% |
| 通信 | 4.6% | 18.6% | 27.1% | 28.0% |
| 注意力计算 | 1.9% | 7.7% | 11.3% | 11.7% |
| Lightning Index | 1.2% | 4.9% | 7.1% | 7.4% |
| MoE（全部） | 0.7% | 3.0% | 4.4% | 4.5% |

融合后，瓶颈迁移至注意力投影（CUBE 受限）—— 这正是 910C 的 2.54 倍 Cube TFLOPS 优势可以充分发挥的领域。

#### 关键洞察

1. **内核融合将 mHC 的 HBM 流量减少约 10 倍**，中间结果保留在寄存器/SRAM 中
2. **融合默认配置使 910C 在 Prefill 上与 H20 具有竞争力** —— 吞吐量差距从 2.57 倍缩小至 1.12 倍
3. **BF16 融合在 FP32 融合+SP 基础上提供额外约 30% 性能提升**
4. **融合后瓶颈迁移至 CUBE 受限的注意力投影** —— 910C 的硬件优势所在
5. **影响在短序列时最为显著**，因为 mHC 占主导；在 256K 时，二次方增长的注意力计算掩盖了 mHC 的影响

#### SP/mHC-SP 对比（融合基线，TP=8，EP=16，DP=2）

| 场景 | 无 SP | 仅 SP | SP + mHC-SP | 加速比 |
|---|---|---|---|---|
| **8K/4K**（910C） | 3,507 ms | 2,492 ms | 1,706 ms | **2.06x** |
| **32K/4K**（910C） | 16,333 ms | 12,244 ms | 9,102 ms | **1.79x** |
| **128K/4K**（910C） | 102,647 ms | 86,264 ms | 73,696 ms | **1.39x** |
| **256K/4K**（910C） | 152,440 ms | 136,058 ms | 123,489 ms | **1.23x** |
| **8K/4K**（H20） | 11,738 ms | 4,398 ms | 4,045 ms | **2.90x** |
| **32K/4K**（H20） | 51,606 ms | 22,233 ms | 20,819 ms | **2.48x** |

SP 单独使用即可在 H20 上带来显著收益（8K 时 2.67 倍），因为大幅削减了 EP AllToAll 通信。在 910C 上，SP 收益较为温和（1.41 倍），因为 EP 带宽较快。mHC-SP 的收益在长序列时递减，因为二次方增长的注意力计算占据主导。

### 5.2 注意力与 KV 缓存分析

#### KV 缓存随序列长度的变化（910C，TP=4，EP=32，DP=8，BS=8）

| 序列长度 | V4 KV 缓存 | 无压缩 | V3 MLA | 相对无压缩节省倍数 | Decode 步时间 |
|---|---|---|---|---|---|
| 1K | 0.03 GB | 0.09 GB | 0.07 GB | 3.4x | 17.9 ms |
| 4K | 0.08 GB | 0.36 GB | 0.29 GB | 4.6x | 19.3 ms |
| 8K | 0.15 GB | 0.72 GB | 0.58 GB | 4.9x | 19.3 ms |
| 32K | 0.55 GB | 2.89 GB | 2.30 GB | 5.2x | 19.4 ms |
| 64K | 1.09 GB | 5.77 GB | 4.61 GB | 5.3x | 19.5 ms |
| 128K | 2.18 GB | 11.54 GB | 9.21 GB | 5.3x | 21.0 ms |

V4 的 KV 压缩相比无压缩注意力节省 3.4--5.3 倍内存，相比 V3 的 MLA 方案节省 3.9--4.2 倍。压缩比在长序列时更优，因为 C4A/C128A 层占主导。

Decode 延迟在不同上下文长度下非常稳定（17.9--21.0 ms）—— KV 压缩有效限制了 Decode 注意力开销。

#### 各层类型 KV 缓存分布

| 序列长度 | Full Attn | C4A | C128A |
|---|---|---|---|
| 8K | 23.0% | 71.6% | 5.4% |
| 32K | 24.3% | 72.8% | 2.9% |
| 128K | 24.6% | 73.0% | 2.4% |

C4A 层在所有序列长度下均主导 KV 缓存使用（71--73%）。2 个全注意力层虽仅占 43 层中的 2 层，但贡献了总 KV 缓存的 23--25%。

#### 压缩与未压缩对比

| 序列长度 | V4 压缩 | V4 未压缩 | V3 MLA | V4 vs V3 |
|---|---|---|---|---|
| 8K | 0.15 GB | 0.72 GB | 0.58 GB | **小 3.9 倍** |
| 32K | 0.55 GB | 2.89 GB | 2.30 GB | **小 4.2 倍** |
| 64K | 1.09 GB | 5.77 GB | 4.61 GB | **小 4.2 倍** |
| 128K | 2.18 GB | 11.54 GB | 9.21 GB | **小 4.2 倍** |

V4 的 KV 压缩是长上下文服务的重大架构优势。在 128K 时，V4 每 rank 仅使用 2.18 GB KV 缓存，而 V3 需要 9.21 GB —— 这意味着是否能放入 64 GB HBM 与是否需要大幅减少 Batch 之间的区别。

#### 注意力计算随序列长度的变化（910C，每层，TP=8，EP=16）

| 层类型 | 8K 注意力% | 32K 注意力% | 128K 注意力% |
|---|---|---|---|
| Full (ratio=1) | 65.1% | 86.1% | 96.1% |
| C4A (ratio=4) | 31.3% | 55.2% | 85.6% |
| C128A (ratio=128) | 31.0% | 39.3% | 54.2% |

全注意力层最先变为注意力主导。在 128K 时，即使 C128A 层也有超过 50% 的时间花在注意力上。这验证了多比率压缩策略的合理性：激进的压缩（C128A）使大多数层在长上下文时保持高效。

## 6. 部署建议

### 6.1 短上下文（8K/4K）—— 对话/编程

| 平台 | Prefill | Decode | P:D | 总 GPU 数 |
|---|---|---|---|---|
| **910C** | TP=8, EP=16, DP=2 (16 GPUs) | TP=4, EP=16, DP=4 (16 GPUs) | 1:1 | 32 |
| **H20** | TP=8, EP=8, DP=1 (8 GPUs) | TP=8, EP=16, DP=2 (16 GPUs) | 1:1 | 24 |

两个平台均能以 1P:1D 比例高效处理 8K 场景。H20 成本效益略优（24 vs 32 个 GPU）。910C 实现了有竞争力的 Prefill 吞吐量（1,656 vs 1,848 tps/gpu）。

### 6.2 中等上下文（32K/4K）—— 文档处理

| 平台 | Prefill | Decode | P:D | 总 GPU 数 |
|---|---|---|---|---|
| **910C** | TP=8, EP=16, DP=2 (16 GPUs) | TP=4, EP=32, DP=8 (32 GPUs) | 1:1 | 48 |
| **H20** | TP=8, EP=8, DP=1 (8 GPUs) | TP=4, EP=16, DP=4 (16 GPUs) | 2:1 | 32 |

在 32K 时，H20 需要 2P:1D，而 910C 保持 1P:1D，因为 Prefill/Decode 平衡更好。910C Decode 吞吐量降至 62 tps/gpu（H20 为 137）—— H20 的带宽优势变得显著。

### 6.3 长上下文（128K/4K）—— RAG/文档问答

| 平台 | Prefill | Decode | P:D | 总 GPU 数 |
|---|---|---|---|---|
| **910C** | TP=8, EP=16, DP=2 (16 GPUs) | TP=4, EP=32, DP=8 (32 GPUs) | 2:1 | 64 |
| **H20** | TP=8, EP=8, DP=1 (8 GPUs) | TP=4, EP=16, DP=4 (16 GPUs) | 4:1 | 48 |

内存约束将 Batch 大小限制在个位数。910C 的 P:D 比例通过内核融合从 4:1 改善至 2:1，总 GPU 数从 96 降至 64。H20 仍需更少的总 GPU 数（48 vs 64）。

### 6.4 超长上下文（256K/4K）—— 全文档分析

| 平台 | Prefill | Decode | P:D | 总 GPU 数 |
|---|---|---|---|---|
| **910C** | TP=8, EP=16, DP=2 (16 GPUs) | TP=4, EP=32, DP=8 (32 GPUs) | 3:1 | 80 |
| **H20** | TP=8, EP=8, DP=1 (8 GPUs) | TP=4, EP=32, DP=8 (32 GPUs) | 14:1 | 144 |

在 256K 场景下，910C 成为 GPU 效率更高的选择（80 vs 144 个 GPU），因为 H20 的 P:D 比例极端（14:1）。需要考虑 256K 上下文的投入是否值得，还是应采用分块策略。

### 6.5 通用指导

1. **内核融合默认开启**（`mhc_kernel_fused=True`）—— 无额外成本即可获得 3--4 倍 Prefill 加速
2. **共享专家重叠默认开启**（`shared_expert_overlapped=True`）
3. **始终启用 SP**（`sp=True`）—— 可免费获得 Norm/激活计算的性能提升
4. **探索 mHC-SP**（`mhc_sp=True`）—— 在融合基础上获得额外的 Prefill 加速
5. **根据网络调整 EP** —— 910C 使用高 EP（EP=16--64），H20 使用低 EP（EP=8--16）
6. **Decode 尽量使用大 Batch** —— 吞吐量随 Batch 大小近乎线性增长，直至内存上限
7. **128K+ 场景必须使用 P/D 分离** —— 混合服务浪费 60--80% 的资源

## 7. 行业启示

### 7.1 KV 缓存管理与分层缓存

长上下文（128K+）从根本上改变了服务格局：
- 由于 KV 缓存内存压力，每 rank Batch 大小缩减至 1--8
- 从 8K 到 128K/256K，Decode 吞吐量下降 10--21 倍
- V4 的 KV 压缩（相比 V3 节省 4.6 倍）至关重要 —— 没有压缩，128K 上下文在合理 Batch 大小下不可行

实际系统需要分层 KV 缓存策略：HBM 用于活跃序列，主机内存或 SSD 用于被换出/抢占的序列。V4 的压缩格式使数据卸载更加高效 —— 需要移动的数据量减少 4.6 倍。

### 7.2 P/D 分离架构

分析证实 P/D 分离对长上下文服务至关重要：
- 8K/4K 时：1P:1D —— 混合服务可行
- 128K/4K 时：2--4P:1D —— 混合服务浪费 60%+ 的资源
- 256K/4K 时：3--14P:1D —— 分离部署不可或缺

Prefill（CUBE 受限）与 Decode（MEM 受限）之间的不对称性意味着最优硬件配置不同：Prefill 受益于高算力和 EP 带宽（910C 的优势），而 Decode 受益于高 HBM 带宽（H20 的优势）。异构 P/D 部署是一种自然的优化方向。

### 7.3 硬件设计权衡

**算力与带宽的矛盾取决于场景：**
- 融合前：H20 的 2.2 倍 HBM 带宽通过 mHC 内存瓶颈主导 Prefill
- 融合后：910C 的 2.54 倍 Cube TFLOPS 驱动 Prefill；瓶颈转向计算
- 这表明软件优化可以从根本上改变硬件竞争力

**HBM 带宽仍是 Decode 的决定性因素：** H20 的 2.2 倍带宽优势稳定转化为 2 倍的 Decode 加速。对于 Decode 密集型工作负载（长输出的聊天机器人），HBM 带宽 > 计算 TFLOPS。

**HBM 容量支持更大 Batch：** H20 的 96 GB 对比 910C 的 64 GB，允许多 50% 的配置放入内存。在 128K/256K 时，H20 可以服务 910C 无法容纳的 Batch 大小。

### 7.4 MoE 的网络带宽需求

EP 带宽是 MoE 模型服务的关键差异因素：
- 910C 的 392 GB/s EP 带宽允许 EP=64 配置仅带来适度的 AllToAll 开销
- H20 的 50 GB/s EP 带宽使 EP=64 配置代价极高（AllToAll 开销高 7.8 倍）
- H20 通过使用较低 EP（EP=8）并在每 rank 放置更多专家来弥补

MoE 模型强烈偏好高带宽互联平台。NVLink 域（8 GPU）在 H20 上高效运行，但跨节点 MoE 开销非常大。

### 7.5 mHC 作为新范式

DeepSeek V4 的 mHC 在架构上具有创新性，此前是主要的性能瓶颈。内核融合的效果具有变革性：
- **未融合**：占 Prefill 时间的 84% —— 最大的单一优化目标
- **融合（默认）**：占 Prefill 的 36% —— 整体加速 4.1 倍
- **融合 + SP + BF16**：3.4% —— 相对未融合整体加速 6.1 倍

经验启示：新型架构组件（如 mHC）最初可能表现为严重的性能瓶颈，但有针对性的内核优化可以将其开销降低一个数量级。在算子层面的硬件/软件协同设计比单纯的硬件规格更为重要。

### 7.6 超长上下文服务

256K 上下文在两个平台上都面临重大挑战：
- Prefill 延迟：34s（910C）/ 66s（H20）—— 对于交互式使用不可接受
- Decode 吞吐量降至 8.5（910C）/ 25.6（H20）tps/gpu
- P/D 比例变得极端：3:1（910C）/ 14:1（H20）
- 总 GPU 预算：80（910C）/ 144（H20）

实际的 256K 服务可能需要分块 Prefill、推测解码或层次化注意力来降低延迟和 GPU 需求。

### 8.2 方法论

- **Roofline 模型**：每个算子时间 = max(cube_time, vec_time, mem_time) + comm_time
- **FLOPs**：矩阵乘法 [M,K]x[K,N] = MxNxKx2
- **BF16**：每元素 2 字节；FP32（mHC）：每元素 4 字节
- **通信**：AllReduce = 2(n-1)/n × vol/BW，AllToAll = (n-1)/n × vol/BW，AllGather = (n-1)/n × vol/BW
- **利用率**：计算 50%，HBM 带宽 80%（两平台相同）
- **内存检查**：weight_per_rank + KV_cache_per_rank <= HBM 容量
- **Decode 近似**：首尾 Decode 步之间的线性插值

**局限性**：通信建模为累加式（不与计算重叠）。采用 Flash Attention 内存模型。Hash 路由层的 load balance factor = 1.0。
